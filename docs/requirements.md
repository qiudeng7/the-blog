我想开发一个博客，最顶层分两个路由，通过屏幕下方的液态玻璃导航，首页和软件开发，首页是个人介绍、我的兴趣和一些项目，软件开发是一个庞大的坐标系，囊括整个软件开发领域我学到的知识，下面我们的重心主要在描述软件开发页面是什么样的。

1. 横轴坐标轴是软件开发流程，比如需求分析、系统设计等，每一个流程对应一个线段而不是一个点，指针移动到上面之后会高亮显示该阶段的主要活动和评价指标，整个内容大致如下矩阵：

| 阶段 | 核心活动 | 核心产出 | 创造的核心价值 | 关键评价维度 |
| :--- | :--- | :--- | :--- | :--- |
| **1. 探索与定义** | 市场调研、商业论证、利益相关者识别 | 商业案例、项目章程、愿景文档 | **方向正确性** | 市场机会大小、投资回报率、战略契合度、利益相关者支持 |
| **2. 可行性分析** | 技术/运营/经济可行性研究，风险评估 | 可行性分析报告、风险评估矩阵 | **风险可控性** | 技术能力匹配、时间与成本估算精度、合规与安全风险 |
| **3. 需求分析** | 需求收集、分析、建模、优先级排序 | 需求规格说明书、用户故事地图、原型 | **业务价值清晰度** | 用户痛点覆盖度、功能完整性、优先级明确性、可测试性 |
| **4. 系统设计** | 架构设计、技术选型、接口设计、数据模型设计 | 系统架构图、技术方案文档、API规范、数据库Schema | **架构健壮性与扩展性** | 模块化与解耦程度、可扩展性、技术栈匹配度、安全性设计 |
| **5. 详细设计与实现** | 模块详细设计、编码、单元测试、代码评审 | 可工作的软件、详细设计文档、单元测试用例 | **功能正确性与内在质量** | 代码可读性/可维护性、测试覆盖率、性能基准、安全编码规范 |
| **6. 测试** | 集成测试、系统测试、性能测试、用户验收测试 | 测试报告、缺陷跟踪列表、质量评估报告 | **质量可靠性** | 缺陷密度/趋势、测试通过率、需求达成率、性能达标率 |
| **7. 部署与发布** | 环境准备、CI/CD流水线执行、上线部署、验证 | 线上运行的系统、发布记录、部署脚本 | **价值交付能力** | 部署成功率、发布频率、平均部署时长、回滚能力与时间 |
| **8. 运营与监控** | 系统监控、日志分析、故障响应、性能调优、容量规划 | 系统仪表盘、运维SOP、事故报告、容量计划 | **系统稳定性与可观测性** | 系统可用性(SLA)、平均恢复时间(MTTR)、监控覆盖率、告警准确率 |
| **9. 维护与演进** | Bug修复、功能迭代、技术重构、文档更新 | 软件新版本、技术债看板、用户反馈报告 | **长期适应性与可维护性** | 变更成功率、技术债水平、演进成本、用户满意度 |
| **10. 终止** | 数据迁移与归档、服务下线、知识转移、资源回收 | 系统退役报告、归档数据、经验教训总结 | **合规与成本收尾** | 数据合规性、资源释放完整性、知识留存度 |


2. 纵轴表示技术的抽象层级，比如从编程语言到库，到框架，到具体的上层应用，这样我们的坐标系就可以通过(x,y)来定位一个区块，在区块内放一个核心技术栈，比如在部署与发布的某一个y轴区块，我可以同时放gitea和docker。
